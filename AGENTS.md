# Agent Instructions

このファイルは、AIチームが各プロジェクトで最初に読む入口です。

## 最初に必ず読むファイル

作業を始める前に、以下を順番に読んでください。

1. `docs/handoff/CURRENT_STATUS.md`
2. `manuals/AI_TEAM_WORKFLOW.md`
3. `manuals/WORKFLOW_RULES.md`

## プロジェクト固有ファイル（存在する場合は必読・拘束力あり）

以下のファイルが存在する場合は、共通ルールに追加して必ず読み、その内容に厳密に従ってください。「参考にする」ものではなく、このファイルと同じ拘束力を持つルールです。

- `docs/PROJECT_RULES.md`（このプロジェクト固有の危険領域・検証条件・Air/CC/Dexそれぞれの追加ルール）
- `docs/handoff/WORKFLOW_RULES.md`（引き継ぎ・出口ゲート・リカバリプロトコルの詳細）

`docs/PROJECT_RULES.md` には、例えば以下のようなプロジェクト固有ルールが書かれていることがあります。実際の内容は毎回ファイルを開いて確認してください。

- Airが設計・実装案を出す前に必要な事前宣言と、Kazumaxの承認待ちルール
- 危険タスクの完全プロセス（Air計画→Dex事前レビュー→CC実装→Dex事後レビュー）の適用範囲
- コード変更時のバージョン管理・ビルド・検証コマンド
- Dexが重点的にレビューすべき領域とP2/P4の保存先

`docs/handoff/CURRENT_STATUS.md` がまだ存在しない場合は、そのプロジェクトは新規開始です。勝手に別プロジェクトの進捗を引き継がず、Cycle 1 から開始してください。

## 共通ルールと同期に関する重要事項

`AGENTS.md`、`CLAUDE.md`、`.cursorrules` はこのプロジェクト専用ファイルではなく、`マニュアル同期.ps1` によって複数プロジェクトへ配布される共通テンプレートです。**このファイル自体にプロジェクト固有のルール（特定の技術用語・コマンド・危険領域名など）を直接書き込まないでください。** 同期が再実行されると無条件に上書きされ、書いた内容は跡形もなく消えます。プロジェクト固有の強いルールは、必ず `docs/PROJECT_RULES.md` や `docs/handoff/WORKFLOW_RULES.md` など、同期対象外のファイルに書いてください。

## 絶対ルール

- コードやファイルを編集する前に、読んだルールと現在の状況を短く要約してください。
- 他プロジェクトのサイクル番号、handoff履歴、GitHub URL、固有設定を引き継がないでください。
- 共通ルールは `manuals/` に置き、プロジェクト固有の進捗や危険ファイル情報は `docs/handoff/CURRENT_STATUS.md` や `docs/PROJECT_RULES.md` に置いてください。
- 指示書、実装報告、レビュー結果は、できるだけ `docs/handoff/` 配下のファイルで受け渡してください。
- 提案や改善案は `docs/proposals/` に保存してください。

## 役割

- Kazumax: 最終判断者、マエストロ。
- Air: 企画、設計、Blueprint作成。
- Dex: 仕様レビュー、CC向け指示書作成、DIFFレビュー。
- CC: 実装、検証、実装報告。

## プロジェクト開始時

新しいプロジェクトでは、`project-template/` の中身をプロジェクト直下へコピーし、`docs/handoff/CURRENT_STATUS.md` のプロジェクト名と開始サイクルを更新してください。

## 共通マニュアル更新ルール

共通ルールを変更する場合、作業中のAIは必ず以下を行ってください。

1. デスクトップの `AI共通マニュアルテンプレ` を更新する。
2. デスクトップの `マニュアル同期.ps1` を実行する。
3. 同期先プロジェクトに変更が反映されたことを確認する。
4. Kazumaxへ「共通マニュアル更新と同期が完了した」と報告する。

プロジェクト内の `AGENTS.md`、`CLAUDE.md`、`.cursorrules`、`manuals/` だけを直接変更して終わってはいけません。共通ルールの本物は常に `AI共通マニュアルテンプレ` です。
