# 開発における絶対ルール (Git & バージョン管理)

**このファイルを読む人**: CC（シーシー / Claude Code）。  
**目的**: CC専用の実装・バージョン管理・コミット運用ルールを定義する。

1. 作業（タスク）が一つ完了するごとに、必ず変更分を Git にコミット（commit）すること。
2. **【超重要】バージョン更新と検証**: コミットする前に、必ず `src/main/resources/application.properties` 内の `app.version` をカウントアップすること。さらに、必ず `.\mvnw compile` を実行し、`target/classes/application.properties` のバージョンも同期して一致していることを検証すること。
3. コミットする際は、必ずコミットメッセージの先頭に更新したバージョン表記（例: `[v1.8.2] 〇〇機能を追加`）を義務付けること。
4. コミット後、必ずリモートリポジトリ（GitHub等）へプッシュ（push）すること。
5. 今後、作業を行う際は、毎回必ずこの「バージョン更新・同期確認・保存・プッシュ」を自律的に完了させてから報告すること。

## 補足（このルールを守るための実務メモ）

- **リモート**: `origin` = https://github.com/kazuma20002106-gif/icehockey-budget-system.git
- **対象ブランチ**: `main`
- **バージョンの管理場所**: `src/main/resources/application.properties` の `app.version`。
  コミット時はこの値とコミットメッセージ先頭の `[vX.Y.Z]` を一致させて更新する。
- **バージョン採番の目安（セマンティックバージョニング）**:
  - パッチ（`v1.6.6 → v1.6.7`）= バグ修正・小さな調整
  - マイナー（`v1.6.x → v1.7.0`）= 機能追加
  - メジャー（`v1.x → v2.0.0`）= 大きな仕様変更・互換性のない変更
- **標準手順**:
  1. `application.properties` の `app.version` を更新（これにより、すべての画面のナビゲーションバーにあるバージョン表記が自動で切り替わり、ユーザーが更新に気づきやすくなります）
  2. 画面のどこを見ればバージョンアップが確認できるかユーザーに案内する（「ブラウザを更新（F5）して、画面左上のバージョン表記が変わったことを確認してください」等）
  3. `git add -A`
  4. `git commit -m "[vX.Y.Z] 変更内容の要約"`
  5. `git push origin main`
- コミット本文の末尾には共著表記を付与する（`Co-Authored-By: Claude ...`）。

## 📥 指示の受け取り方（自律行動ルール）
Kazumaxから「指示が出たよ」「差し戻しがあったよ」と短い合図があった場合、パスを聞き返すのではなく、**自ら以下のフォルダを検索して最新の `.md` ファイルを読み込み、作業を開始**してください。
- 初回の作業指示：`docs/handoff/P2_Dex_to_CC/`
- 差し戻し（NG）指示：`docs/handoff/P4_Rollback/`

## 🚨 作業完了時の出力フォーマット（絶対ルール）
実装・コンパイル・コミットが完了したら、Kazumaxのチャット画面には**絶対に長文の報告やコードの解説を出力しないこと**。
人間のコピペ労力をなくすため、以下のルールで「ファイル経由（Handoff Protocol）」でDex（P4）に完了報告を渡すこと。

1. Dex向けの詳しい実装完了報告（変更内容、ファイル一覧など）は、すべて `docs/handoff/P3_CC_to_Dex/cycle_〇.md` というファイルを作成（または上書き）して、そこに保存すること。
2. チャット画面（Kazumaxへの返答）には、以下の要素を順番に出力すること。

- **完了報告**: `実装が完了し、docs/handoff/P3_CC_to_Dex/cycle_〇.md に報告書を作成しました！`
- **超簡潔な要約**: （例「ツールチップと幅を直しました！」など一言）
- **💡 ＋αの提案について**: （提案内容を短く書き、かつ、全く同じ内容を `docs/proposals/CC_cycle_〇.md` にも保存したことを伝える）
- **🎯 現在のステータス**: 出力の最後に現在の状況と「次に誰の何を待っているか」を明記する。
  - （例：`現在のステータス：実装完了。Dex（P4）からの差し戻し or 新たな指示（P2）待ち`）

### ⏩ 次の担当（Dex）への合図（コピペ用）
KazumaxがそのままコピーしてDexに投げられるよう、**以下の文言だけを単独のコードブロック**にして出力してください。見出しはコードブロックに入れないこと。

```text
CCの実装が終わったよ。最新のファイルを読んでDIFFレビュー（P4）をして！
```
   提案内容（技術的な改善案やルール更新案）は、Kazumaxが見れるようにチャットに出力するとともに、必ず `docs/proposals/CC_cycle_〇.md` というファイルにも同じ内容を書き込んで保存すること。Airが後でまとめてレビューします。
