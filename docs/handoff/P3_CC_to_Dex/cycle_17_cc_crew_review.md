# Cycle 17 UX改善 CCクルーによる補助レビュー

Dexが多忙のため、Kazumaxの依頼によりCCの下で「CCクルー」というサブエージェントを起動し、Cycle 17実装（commit `2b84a32`／`f99ee31`）をP3報告書の主張を鵜呑みにせず独立に検証しました。**Dex(P4)の正式レビューを代替するものではありません**が、参考情報として共有します。

対象commit: `2b84a32`（実装）／`f99ee31`（docsのみ）。差分レンジ `eedbb7b..f99ee31` を全読、`.\mvnw.cmd -q -DskipTests compile` 成功、`app.version` はsrc/target双方 `v2.4.9` 同期を確認済み。

## 1. 総合判定

**問題なし（軽微な観察事項のみ）。** 金額計算の正確性・二重計上防止・DBスキーマ安全性・一括処理の安全性・HTMLフォーム妥当性のいずれも、実コードで独立検証した結果、Kazumax代表の絶対方針（合算の正確性）を損なう欠陥は見つかりませんでした。P3報告書の主張はすべて実コードで裏付けが取れました。修正必須(blocking)の問題はありません。

## 2. 発見した問題点（いずれも修正必須ではない）

- **[軽微] Cycle 8由来のALTER文が依然として無効構文のまま**（`schema.sql:132-134`）。新規DB構築時に`accommodation_nights`等が作られず起動時エラーになるリスクが残存。`docs/proposals/CC_cycle_17_schema_alter_audit.md`にCCが起票済み。Phase2 VPS本番DB構築前の別サイクル対応を推奨（提案内容は妥当と判断）。
- **[情報] 一括「2-2プレビュー」は選択活動を区分横断で合算**（既存`ExportController.preview`の挙動をそのまま再利用。今回の新規バグではなくlegacy挙動そのまま）。
- **[情報] `error`表示が`param.error`とmodel属性`error`の2系統混在**（`list.html`）。動作影響なし、スタイル不統一のみ。
- **[情報] 複製時の複数Expense検出クエリが`ActivityController.duplicate`と`ProjectService.duplicateProject`で二重発行**。規模的に実害なし。
