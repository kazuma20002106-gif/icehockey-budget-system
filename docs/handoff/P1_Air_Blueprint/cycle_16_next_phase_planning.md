# Cycle 16 Blueprint: 複数Expenseの完全対応（または運用制限）と本番公開に向けた準備

これまでの厳密なバグ修正・監査により、システムの計算ロジックや画面とExcelの不一致はすべて解消されました。
いよいよ「依頼主が使えるようにする（サーバー構築）」フェーズが目前に迫っていますが、その前に**「最後の未解決リスク」**をどう扱うか方針を決定するためのBlueprintです。

---

## 1. 残された最後のリスク：編集画面での「複数Expense」のデータ消失

Cycle 13で「1人の参加者が複数の経費（Expense）を持っている場合、プレビューやExcelで全件合算する」という修正を行いました。これにより出力機能は完璧になりました。
しかし、**経費の入力・編集画面（`/activity/{id}/edit`）**は現在も「参加者1人につき経費入力欄は1行のみ」というUIのままです。

**【現在の危険な挙動】**
もしデータベース上に「1人の参加者に2つの経費」が登録されている状態で編集画面を開くと、1件目しか画面に表示されません。そのまま「保存」ボタンを押すと、裏側でデータが一度全削除されてから画面のデータ（1件目）だけで再登録されるため、**見えていなかった2件目以降の経費データが永久に消滅（物理削除）**してしまいます。

---

## 2. Kazumax様への確認事項（方針決定）

このリスクに対する対応方針を、以下の3案からご選択ください。

### 案A: 編集画面も「複数Expense」に完全対応する（推奨・高難易度）
- **方針**: 編集画面のUIを改造し、「参加者1人に対して、経費の入力行を複数追加できるボタン」を新設します。
- **メリット**: 実際に複数回の移動や宿泊があった場合でも、そのまま詳細に入力・管理できる完璧なシステムになります。
- **デメリット**: 画面のJavaScriptとHTMLを根本的に改造する必要があり、CCの実装難易度が高く時間がかかります。

### 案B: 運用を「1参加者1Expense」に制限する（中難易度・現実的）
- **方針**: 画面のUI改造は行わず、「1人につき保存できる経費データは1行のみ」という制限（バリデーションや強制的な合算処理）をシステム側にかけます。
- **メリット**: もし複数回の移動があった場合は「ユーザーが電卓で合算した金額を、1つの欄に手入力する」という運用ルールにすることで、システムの複雑化を避けつつデータ消失事故を防げます。

### 案C: システム改修は完了とし、本番公開（サーバー構築）へ移行する
- **方針**: 依頼主にはマニュアル等で「1人につき1行で合算して入力してください」と伝え、これ以上のシステム改修（Cycle 16）は行いません。このまま次の大目標である**「Linux(VPS)でのサーバー構築・デプロイ作業」**へ移行します。
- **メリット**: 最速で依頼主にツールを納品できます。（サーバー構築は次回の居酒屋HPプロジェクトの基盤作りにも繋がります）。
