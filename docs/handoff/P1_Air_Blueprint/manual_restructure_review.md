# 📍 チームマニュアル再整理計画（Dex案）に対するAir(PM)レビューとCC向け実装方針

## 1. Dex案へのPM視点レビュー

Dexが作成したマニュアル分割案（`AGENTS.md`, `WORKFLOW_RULES.md`, `PROJECT_RULES.md`, `CURRENT_STATUS.md` の4層構造）について、PM視点から安全性・運用しやすさ・不足ケースをレビューしました。

### ✅ 安全性（最高）
`AGENTS.md`（普遍的ルール）と `PROJECT_RULES.md`（案件固有の危険領域）を分離するアプローチは非常に優れています。これにより、AIが「このプロジェクト特有の地雷」を明確に意識できるようになり、同時に他のプロジェクトへチーム基盤を展開する際にも `AGENTS.md` をそのまま使い回せるようになります。

### ✅ 運用しやすさ（高い）
起動時の合図文（Bootstrap Text）に「`PROJECT_RULES.md` がある場合は読む」という条件付きの記述を入れることで、新チャット開始時のコンテキスト復元が確実かつ柔軟に行えるようになります。

### ⚠️ 補足・追加すべきケース（不足ケース）
Dexの案をベースにしつつ、以下の微調整をCCに指示します。
1. **越権行為のリカバリ対象の拡大**: Dex案の「Air Overreach Rule（Airの越権行為ルール）」について、Airだけでなく「役割を逸脱したすべてのAIによる予期せぬ破壊的変更（CCの誤削除なども含む）」に対する汎用的な『リカバリプロトコル』として `WORKFLOW_RULES.md` に記載すること。
2. **CURRENT_STATUS.md の更新責任の明確化**: 「フェーズの移行（例：実装完了、レビュー依頼、承認待ち）を発生させたAIが、必ずCURRENT_STATUS.mdの現在地を書き換える」というルールを `WORKFLOW_RULES.md` の出口ゲート（チェックリスト）に明記すること。

## 2. CC向けの実装方針（修正指示）

CCは、以下の手順でマニュアルの再編成・更新を実行してください。今回はコード（Javaや設定ファイル）の変更はありません。

1. **`docs/PROJECT_RULES.md` の新規作成**
   - Dex案のセクション2-Cに記載された `budget-system` 固有の危険領域（`src/main/java/`, `mapper/`, `schema.sql`, `*.xlsx`, `ExcelExportService.java` など）を記載する。
   - 金額計算やExcel出力に関するリスクもこのファイルに定義する。

2. **`AGENTS.md` の編集**
   - プロジェクト固有のファイルパス（`ExcelExportService.java` など）を削除し、普遍的な役割定義とルールに絞る。
   - 第0章の【タスク重要度・事前宣言】は残しつつ、起動合図文のフォーマットをDex案の最新版（`WORKFLOW_RULES.md` と `PROJECT_RULES.md` を含むもの）に差し替える。

3. **`docs/handoff/WORKFLOW_RULES.md` の編集**
   - Airの越権行為リカバリルールを「AIの予期せぬ破壊的変更時のリカバリプロトコル」として汎化して記載する。
   - `CURRENT_STATUS.md` の更新責任者を明確化する。

4. **`docs/handoff/CURRENT_STATUS.md` の整理**
   - 過去の長すぎる履歴や関係ない提案を削除し、現在の「マニュアル整備タスク」のみに絞ったコンパクトな形に整理する。

5. **完了報告とコミット**
   - 以上のマークダウンファイルの編集を完了したら、`docs/handoff/P3_CC_to_Dex/` に報告書を保存し、DexのP4レビュー（マニュアル変更差分の監査）へ引き継ぐこと。
