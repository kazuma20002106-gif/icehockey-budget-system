# Airへの運用ルール追加依頼: 危険タスク判定ゲート + 長文は裏ファイル運用

## 目的

今後のAIチーム運用では、基本形を以下に統一します。

```text
通常タスク:
Kazumax → Air(P1/P2統合) → CC → Dexレビュー → Kazumax

危険タスク:
Kazumax → Air(P1/P2草案) → Dex事前監査 → CC → Dexレビュー → Kazumax
```

Kazumaxが毎回「これは危険だからDexを挟もう」と判断しなくて済むように、Airが自律的に危険タスク判定を行い、必要な場合はCCへ直行せずDex事前監査へ回す運用にしてください。

## 反映してほしいファイル

Airが妥当と判断する範囲で、以下のマニュアル・現在地ファイルに反映してください。

- `AGENTS.md`
- `AI_TEAM_WORKFLOW.md`
- `.cursorrules`
- `CLAUDE.md`
- `docs/handoff/CURRENT_STATUS.md`

すべてに重複して長文を書く必要はありません。
入口となるマニュアルには要約を書き、詳細は1箇所に集約して参照させる形でもOKです。

## 追加したいルール1: P1/P2統合運用

今後は、Airが計画(P1)とCC向け実装指示(P2相当)をまとめて作る運用を基本にしてください。

### 理由

- P1からP2への橋渡し待ちを減らせる。
- Airの設計意図がCCへ直接伝わる。
- Kazumaxのコピー回数が減る。
- Dexは事前翻訳ではなく、レビュー・診断・安全監査に集中できる。

## 追加したいルール2: 危険タスクはDex事前監査を必須にする

Airはタスクを受け取ったら、CCへ渡す前に危険タスクかどうかを判定してください。

危険タスクの場合は、CCへ直行せず、以下を行ってください。

1. 裏ファイルにDex事前監査用のP1/P2草案を書く。
2. `CURRENT_STATUS.md` をDex担当に更新する。
3. 表チャットには、KazumaxがDexへそのまま渡せる短い合図文だけを出す。

## 危険タスクの定義

以下のいずれかに該当する場合は、原則としてDex事前監査を挟んでください。

- DB構造・保存処理・データ移行に触る。
- 金額計算、集計、精算、請求、支払いに触る。
- Excel/PDF/帳票出力に触る。
- 参加者名簿、個人情報、認証、権限に触る。
- ファイル削除、移動、上書き、ロールバック、自動復旧に触る。
- 外部プロセス起動、CLI、API、課金、ネットワーク通信に触る。
- 自動化ランナー、監視、PAUSE、manifest、done.json、processed.logに触る。
- 広範囲リファクタ、共通部品、複数画面にまたがる変更。
- 仕様が曖昧で、実装者の判断が多くなりそうな変更。
- 失敗した時にデータ破損・課金・大量変更・復旧困難につながる変更。
- Air自身が「通常タスクか危険タスクか迷う」と感じた変更。

迷った場合は、安全側に倒してDex事前監査へ回してください。

## 通常タスクの定義

以下のような軽量タスクは、Air → CC直行でOKです。

- 文言修正。
- 軽いUI調整。
- 単一画面の小さい表示変更。
- CSS微調整。
- 既存仕様に沿った小さなバリデーション追加。
- テストやドキュメントの軽微な追記。
- 失敗しても簡単に確認・修正できる変更。

## 追加したいルール3: 「これも危険だったね」フィードバックループ

Kazumaxは危険判定を毎回できるとは限らないため、Air・Dex・CC側で危険判定ルールを育ててください。

作業中またはレビュー後に「これは危険タスクとして扱うべきだった」と判断した場合は、以下を行ってください。

1. 該当理由を作業報告またはレビューに明記する。
2. 必要に応じてマニュアルの危険タスク定義に追記する。
3. `CURRENT_STATUS.md` または次の合図文でKazumaxに簡潔に報告する。

報告例:

```text
今回の作業で「〇〇」は危険タスク扱いにした方がよいと判断したため、危険タスク定義に追記しました。
次回から同種の作業はDex事前監査を挟みます。
```

## 追加したいルール4: 長文は裏ファイル、表チャットは合図文

Kazumaxのチャット欄を長文で埋めないため、以下を標準化してください。

### 裏ファイルに書くもの

- 詳細なP1/P2。
- Dex事前監査用の論点。
- CCへの具体的実装指示。
- 修正理由。
- リスク一覧。
- テスト観点。
- 失敗時の復旧手順。
- マニュアルに反映すべき長文ルール。

### 表チャットに出すもの

- 現在地の短い要約。
- 次の担当者名。
- 読むべき裏ファイルのパス。
- Kazumaxがそのままコピペできる合図文。

### 判断基準

以下のどれかに該当する場合は、表チャットではなく裏ファイルに書いてください。

- 10行を超える。
- コマンドや手順が複数ある。
- リスク・禁止事項・例外条件が含まれる。
- 後から別チャットで再開する可能性がある。
- Air/CC/Dexの複数人が読む必要がある。

## Airが危険タスクと判断した時の表チャット例

```text
今回の内容は危険タスクに該当するため、CCへ直行せずDex事前監査を挟みます。

Dex向けの確認依頼を以下のファイルにまとめました。
- docs/handoff/P1_Air_Blueprint/xxxx.md

Kazumaxは以下をDexへコピペしてください。

Dex、Airが危険タスク判定によりCC直行を止めて、事前監査用の草案を作成したよ。
最新のファイルを読んで、安全性・実装指示の明確さ・CCへ渡してよいかをレビューして！
```

## Airが通常タスクと判断した時の表チャット例

```text
今回の内容は通常タスクと判断したため、AirからCCへ直行します。

CC向けの実装指示は以下のファイルにまとめました。
- docs/handoff/P1_Air_Blueprint/xxxx.md

Kazumaxは以下をCCへコピペしてください。

CC、Airが通常タスクとしてP1/P2統合指示を作成したよ。
最新のファイルを読んで実装して！
作業が終わったらCURRENT_STATUS.mdを更新して、Dexレビュー向けの合図文を出してね。
```

## Dexの推奨

このルールは完全自動化より優先して導入する価値があります。

理由は、Kazumaxの判断負担を減らしつつ、危険な変更だけDexを挟めるためです。
通常タスクは速く進み、危険タスクは安全に止められます。
