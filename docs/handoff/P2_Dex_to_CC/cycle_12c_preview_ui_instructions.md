## 0. 12Cの設計強化ポイント

12Cは「見やすくする作業」ですが、金額を扱うため油断しないでください。

方針:

- **真実の金額は12A/12Bの集計ヘルパーから取る。**
- **UI専用の再計算を作らない。**
- **Excelセル座標を12Cで増やさない。**

- **年度末専用の導線として追加する。**

## 1. 出力パラメータ入力

年度末決算ファイル出力前に、以下を入力/確認できるようにしてください。

| 項目 | 初期値 | 備考 |
|---|---|---|
| 提出日 年 | 現在日から算出 | 令和年または西暦から既存仕様に合わせる |
| 提出日 月 | 現在月 | 数字 |
| 提出日 日 | 現在日 | 数字 |
| 団体名 前半 | `宮崎県アイスホッケー` | 12Aの2-1セルに合わせる |
| 団体名 後半 | `連盟` | 12Aの2-1セルに合わせる |
| 代表者職・氏名 | `会長　黒木 誠一郎` | 必要なら既存原本表記に合わせる |

注意:

- 入力値はプレビューとダウンロードで同じものを使う。
- 画面で入力した値がExcelの様式2-1に反映されること。
- ただし、2-1セル座標は12Aの指示書を優先する。

実装方法は、既存構成に合わせて以下のどちらかでよいです。

- `/export/year/preview` のような年度末専用プレビュー画面を作る。
- 既存 `/export/preview` を拡張する。

Dexとしては、既存の通常出力プレビューを壊さないため、年度末専用ルートを推奨します。

### 推奨ルート

既存の `/export/preview` を大きく作り替えず、年度末専用に以下を追加する形を推奨します。

- `GET /export/year/setup`
- `POST /export/year/preview`
- `POST /export/year/download`

`GET /export/year/setup` で条件と提出情報を入力し、`POST /export/year/preview` でタブプレビュー、`POST /export/year/download` で同じ条件のExcelを出力します。

既存の `GET /activity/export/year` がある場合は、互換性を壊さず、年度末専用UIへの入口を追加するだけにしてください。

### 入力値オブジェクト

Controller内では、以下のようなフォームオブジェクトまたは同等の受け渡しを使ってください。

- `year`
- `budgetTypeId`
- `month`
- `targetCategory`
- `projectName`
- `submitYear`
- `submitMonth`
- `submitDay`
- `organizationNamePart1`
- `organizationNamePart2`
- `representativeTitleAndName`

hidden inputで次画面へ渡す値も、この項目名に揃えてください。
プレビューとダウンロードで名前がズレると、Excelに違う値が入る危険があります。

## 2. 年度末プレビューのタブ構成

年度末プレビューでは、以下のタブを表示してください。

- `表紙(2-1)`
- `まとめ(2-2)`
- `選手強化(2-2-1)`
- `トップ(2-2-1)`
- `ふるさと(2-2-1)`
- `変更報告(2-3)`

12Bでトップチームの2-3自動書込をスキップしている場合は、変更報告タブにもその注意を表示してください。

### 表示順

タブ順はExcelの主要シート順に合わせてください。

1. 表紙(2-1)
2. まとめ(2-2)
3. 選手強化(2-2-1)
4. トップ(2-2-1)
5. ふるさと(2-2-1)
6. 変更報告(2-3)

タブの初期表示は `表紙(2-1)` にしてください。
金額確認を急ぐ場合でも、提出情報の確認を飛ばしにくくするためです。

プレビューは「Excel出力前の確認」であり、完全なExcel再現でなくてよいです。

### 表紙(2-1)

- 団体名

### 2-2-1各タブ

- 交通費
- 宿泊費
- 旅行雑費

- 借用料
- 報償費
- 需用費
- 役務費
- 対象外経費
- 支出合計

補助金区分で対象外の費目は、空欄ではなく「対象外」または「出力対象外」と表示してください。

### 変更報告(2-3)

- 内示額
- 決算額
- 差額
- 上部の変更理由・移動額欄は手入力運用であること
- トップチームが2-3へ自動書込されない場合、その理由

## 4. API設計

Air草案の `/api/export/preview-yearly` は採用してよいです。
ただし、返す値はExcel用の真実値と同じ集計ヘルパーから作ってください。

Dex推奨は、まずサーバーサイドレンダリングで年度末プレビューを完成させることです。
API化は、画面実装が複雑になりすぎない場合だけ採用してください。
APIを作る場合でも、12CでDB保存やExcel生成をしない原則は変えません。

推奨:

- `GET /api/export/preview-yearly?year=2025`
- 必要に応じて `budgetTypeId`, `month`, `targetCategory`, `projectName` も受ける

返却JSONの例:

```json
{
  "form21": {},
  "form22": {},
  "form221": {
    "senshu": {},
    "top": {},
    "furusato": {}
  },
  "form23": {},
  "warnings": []
}
```

注意:

- API内でDB保存しない。
- API内でExcelファイルを生成しない。
- プレビュー専用の集計とExcel出力用の集計を別々に作らない。
- 金額はJSONでも数値として返す。文字列化は画面側で行う。
- 警告は `warnings` 配列に入れ、画面上で必ず表示する。
- APIを追加した場合も、P3報告書にリクエスト例とレスポンス例を載せる。

## 5. ダウンロード導線

プレビュー画面の最後に「この内容でExcelをダウンロード」ボタンを置いてください。

ダウンロード時に送る値:

- 年度/検索条件
- 提出日
- 団体名
- 代表者職・氏名

プレビュー時とダウンロード時で条件がズレないよう、hidden inputまたはセッションではなく、フォーム送信値として明示的に渡してください。

注意:

- プレビュー画面で表示した提出日・団体名・代表者情報を、そのままダウンロードPOSTへ渡す。
- ダウンロード時に現在日で再計算し直さない。
- 年度や検索条件も同じ値を渡す。
- ブラウザの戻る操作後でも、ユーザーが見た条件と出力条件がズレないようにする。

- `ExcelExportService.java` の引数受け渡し部分のみ
- `templates/export/year_setup.html`

- `templates/layout.html`

- 12A/12Bで確定したセル座標

12CでExcelテンプレートを触る必要が出た場合は、そこで停止してDexへ戻してください。

## Dexメモ

12Cは便利機能ですが、金額の真実を作る場所ではありません。
UIで見えている数字とExcel出力の数字がズレるのが一番危険です。

そのため、12Cでは以下を絶対ルールにします。

- 集計ロジックを二重管理しない。
- Excelセル座標を再設計しない。
- DBスキーマを変えない。
- 12A/12Bの出力結果を壊さない。

CCはこの範囲で実装し、P3をDexへ戻してください。
