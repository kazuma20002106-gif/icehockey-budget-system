# Cycle 9 実装計画書（P1/P2統合指示書）: 情報量に応じたUI幅と余白の最適化

## 📌 背景と目的
活動一覧など情報量が多い画面は横長・広めのUIが適していますが、現在の「名簿管理」や「提出データ出力」のような情報量が少ない画面まで全体が横長に引き伸ばされ、無駄な余白が多く視線移動が大変な状態（スカスカ）になっています。
本サイクルでは、共通のレイアウトを壊すことなく、画面の情報量に応じた「適正な最大幅（max-width）」や「入力欄ごとの幅調整」を行い、より使いやすい業務ツールへのUI/CSS最適化を実施します。

## ⚠️ Airによる危険度判定：【通常タスク（低リスク）】
**本タスクは「通常タスク（CCへ直行可能）」と判定します。**
理由：
1. DBの変更や金額計算・Excel出力のバックエンドロジックには一切触れないため、システム破壊やデータ欠損のリスクがゼロ。
2. 共通の `layout.html` や `style.css` への破壊的変更を行わず、対象ページ（名簿・出力）専用の局所的なクラス追加で対応するため、活動一覧などの他画面が崩れるリスクを隔離できる。

> Dexによる事前監査は不要と判断します。この指示書をそのままCCへ渡し実装させます。実装完了後の **P4 (diffレビュー)** はDexにて必ず実施してください。

## 📝 Proposed Changes (CC向け 実装指示)

### 1. 対象ファイルの限定
以下のファイルのみを編集対象とすること。これ以外のファイル（特に `activity/form.html` や `activity/list.html`）には影響を与えないこと。
- `src/main/resources/templates/members/index.html`
- `src/main/resources/templates/export/index.html`
- `src/main/resources/static/css/style.css` (※専用クラスの追加のみ許可)

### 2. 専用ラッパークラスによる最大幅の制限
情報量が少ないページが超ワイドモニターで間延びするのを防ぐため、`style.css` に新しい局所用のクラスを追加し、対象の2画面のメインコンテンツ全体をラップする。

**【style.cssへの追加例】**
```css
/* 情報量が少ない画面用の適正幅コンテナ */
.content-narrow-wrap {
    max-width: 900px;
    margin: 0; /* 左寄せ。中央寄せにしたい場合は margin: 0 auto; でも可 */
}
```

**【members/index.html, export/index.html への適用】**
- 画面タイトル（`<h1>`）から下のコンテンツ全体を `<div class="content-narrow-wrap">` で囲む。

### 3. 名簿管理（members/index.html）のフォームと列幅の最適化
- **新規追加フォームの最適化**
  - 現在の `col-md-3`, `col-md-2` などの均等分割をやめ、中身の文字量に合わせた幅にする。
  - 例：年齢入力欄（`type="number"`）は `max-width: 80px;` 等、極力小さくする。種別（セレクトボックス）も中身の文字数に合わせる。氏名と主な出発地点のみ適度な幅を持たせる。
  - グリッドには `col-auto` や明示的な `style="max-width: XXpx;"` を用いて、フォーム全体が横に引き伸ばされすぎないように左寄せでコンパクトにまとめる。
- **登録メンバー一覧（テーブル）の列幅最適化**
  - 氏名、年齢、種別、主な出発地点の入力欄の `width` や `max-width` を、実際に入る文字数ベースで適正化する。
  - 特に「年齢」の `input type="number"` のセルが横に広がりすぎないようにする。

### 4. 提出データ出力・集計（export/index.html）の最適化
- カード（出力事業の選択）およびテーブルが画面いっぱいに間延びしないよう、上記 `content-narrow-wrap` を適用して幅を抑制する。
- テーブル内の列（ID、事業名、期日、場所）についても、事業名には十分な幅を持たせつつ、全体としてスカスカ感をなくす。
