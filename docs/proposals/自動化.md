# Air・CC・Dex連携の自動化案

作成日: 2026-06-19  
作成者: Dex  
目的: 現在のP1・P3・P4ファイル運用を維持したまま、Kazumaxが担当間のメッセージを手動送信する最後の工程まで自動化できるか検討する。

## 1. 現在の運用

現在、引き継ぎ情報そのものはほぼ自動化できている。

1. AirがP1指示書を所定のフォルダーへ作成する。
2. Kazumaxが、画面に表示された短い合図をコピーしてCCへ送る。
3. CCはP1を読み、実装後にP3報告書を作成する。
4. Kazumaxが、画面に表示されたレビュー依頼をDexへ送る。
5. DexはP1・P3・git差分を読み、OKまたはNGを判定する。
6. NGの場合はP4修正指示を作成し、KazumaxがCCへ短い合図を送る。
7. OKの場合はAirへ次サイクル開始の合図を送る。

各担当が詳細をファイルへ記録し、次の担当はそのファイルを読むため、Kazumaxは長文を転記する必要がない。残っている手作業は、各AIのテキストボックスへ短い合図を貼り付けて送信する部分である。

## 2. 自動化の目標

Kazumaxを伝言係から外し、次の流れを自動化する。

```text
AirがP1完成
  ↓
CCが自動でP1を読み実装
  ↓
CCがP3完成
  ↓
Dexが自動でレビュー
  ↓
NGならCCへ自動差し戻し
OKならKazumaxへ完了通知
```

Kazumaxの関与は、最初の開始承認、危険な変更の承認、最終確認を基本とする。

## 3. 3サービス体制を維持する理由

Air・CC・Dexを1つのサービスへ統合すれば連携は簡単になるが、現在の3サービス体制には重要な利点がある。

- 異なるモデルの視点により、同じ思い込みによる見落としを減らせる。
- Airの設計、CCの大規模実装、Dexのレビューという得意分野を使い分けられる。
- 実装者とレビュアーを異なるモデルにでき、レビューの独立性が高い。
- 利用制限を複数サービスへ分散できる。
- 1サービスが制限や障害で止まっても、他の担当で調査や準備を続けられる。

特に、過去にCC単体で大規模なコード改修を行った際、早い段階で利用制限へ到達した経験がある。サブエージェントも同じサービスの契約枠を消費する可能性があるため、単一サービスへの統合は利用制限の面で不利になる恐れがある。

現時点の推奨は、3サービス体制を維持し、担当間の送信操作だけを自動化するハイブリッド方式である。

## 4. APIを使わない自動化案

APIを利用すると各Proプランとは別に従量課金が発生する可能性がある。そのため、まず契約ログイン済みのアプリまたはCLIを利用する方法を検討する。

### 案A: 契約ログイン対応CLIによる連携

各サービスが、契約アカウントでログインしたCLIを外部プログラムから非対話実行できる場合の第一候補。

- ローカル監視プログラムがP1・P3・P4の完成状態を検知する。
- 担当サービスのCLIへ短い開始指示を渡す。
- 担当は既存の指示書を読み、作業と報告書作成を行う。
- 完了ファイルの更新を次工程の開始条件にする。

利点:

- APIキーを使わず、契約枠を利用できる可能性がある。
- 画面位置に依存しないため比較的安定する。
- 完了コードやログを取得しやすい。

未確認点:

- 各サービスの契約プランで非対話実行が許可・提供されているか。
- 非対話実行も契約枠として処理され、API従量課金にならないか。
- セッション再開、ファイル編集、コマンド実行を自動で行えるか。

### 案B: Windows画面操作による連携

正式なCLI入口がないサービスは、AutoHotkey v2、Power Automate Desktop、Windows UI Automationなどで契約済みアプリを操作する。

- 対象アプリを前面へ出す。
- 決められたスレッドまたは画面を選ぶ。
- 合図をテキストボックスへ貼り付ける。
- 送信操作を行う。
- 完了判定は画面ではなく、P1・P3・P4のファイル更新で行う。

利点:

- API料金を追加せず、現在のアプリと契約を維持できる。
- 現在の運用をそのまま自動送信へ拡張できる。

欠点:

- アプリ更新、ウィンドウ名、フォーカス、画面配置の変更で壊れやすい。
- 誤送信防止と停止機構が必須。
- 各サービスの利用条件上問題がないか、正式運用前に確認が必要。

### 案C: API連携

最も安定しやすいが、各Proプランとは別の従量課金になる可能性が高いため、現時点では最終候補とする。

## 5. Maestro Runner案

ローカルPC上に小さな中央制御プログラムを置く。

### 主な役割

- 指示書・報告書のファイル更新を監視する。
- ファイルが保存途中でないことを確認する。
- 現在のサイクルと担当を状態ファイルで管理する。
- CLIまたは画面操作で次担当へ合図を送る。
- OK・NGに応じて次工程を決める。
- 送信内容、時刻、対象、結果をログへ残す。
- 異常時は停止してKazumaxへ通知する。

### 状態ファイル例

```yaml
cycle: cycle_9
phase: p1_ready
approval: approved
attempt: 1
max_attempts: 3
p1_file: docs/handoff/P1_Air_Blueprint/cycle_9.md
p3_file: null
p4_file: null
last_processed_hash: null
```

### 必須の安全装置

- 同じ更新を二重処理しない。
- ファイルサイズと更新時刻が一定時間変化しないことを確認してから読む。
- サイクル番号が一致しない古い指示書を無視する。
- NG差し戻しの自動反復は最大3回などに制限する。
- DB削除、大量削除、外部送信、公開操作などはKazumaxの承認待ちにする。
- エラー時や判定不能時は勝手に次へ進まず停止する。
- 緊急停止用の `PAUSE` ファイルまたは停止ボタンを用意する。
- 各担当の同時編集を避け、工程ごとに編集権を切り替える。

#### Q1. 外部プログラムから非対話実行できるか

**可能（確認済み）**。

`claude.exe` は `-p` / `--print` フラグを公式にサポートしている。

```powershell
$claude = "C:\Users\kazum\AppData\Local\Packages\...\claude.exe"
& $claude --print "差し戻しがあったよ。P4の修正指示を読んで直して！"
```

ヘルプより: *"Print response and exit (useful for pipes)"*。

---

#### Q2. 非対話実行時も契約枠が使われ、API従量課金を発生させないか

**おそらく可能だが要検証**。

Claude Code デスクトップアプリは Claude.ai へのOAuth認証で動作している。`--print` モードでもデスクトップアプリと同じ認証を引き継ぐ設計のため、Pro/Maxサブスクリプション枠で処理される可能性が高い。  
ただし `ANTHROPIC_API_KEY` 環境変数が設定されている場合は従量課金APIへ切り替わる。

**要検証**: 初回は短いプロンプトで `--print` 実行し、請求画面でAPI課金が発生していないことを確認すること。

---

#### Q3. claude コマンドがPCのどこかに存在するか

```
C:\Users\kazum\AppData\Local\Packages\Claude_pzs8sxrjxfjjc\LocalCache\Roaming\Claude\claude-code\2.1.181\claude.exe
```

#### Q4. プロンプト文字列または指示ファイルを引数・標準入力で渡して作業を開始できるか

**可能（確認済み）**。複数の渡し方が使える。

```powershell
# 引数として直接渡す
& $claude -p "差し戻しがあったよ。P4の修正指示を読んで直して！"

```

`--input-format` オプションもあり、構造化入力も可能。

#### Q5. 既存のClaude Codeセッションを指定して再開できるか

**部分的に可能**。

```powershell
& $claude --resume <session_id>   # セッションIDで指定再開
& $claude --continue              # 最新のセッションを再開
```

ただし任意のセッションIDを事前に知っている必要がある。Maestro Runnerが前回の実行時に出力されたセッションIDを状態ファイルへ保存しておけば指定再開が可能。

---

#### Q6. 作業完了を終了コード・フック・出力ファイルなどで外部へ通知できるか

**可能（確認済み）**。

- **終了コード**: 成功=0、失敗=非0。`$LASTEXITCODE` で判定できる。
- **標準出力**: `--print` モードの出力をパイプやリダイレクトでキャプチャできる。
- **Claude Code Hooks**: `Stop` イベントにシェルコマンドを登録できる。設定ファイル（`.claude/settings.json`）で設定可能。
- **ファイル書き込み**: CCが作業完了時にP3報告書を書くのが現行の方式で、Maestro Runnerはこのファイルの完成を検知すれば良い。

---

#### Q7. 非対話モードで、ファイル編集・git差分確認・ビルド・P3報告書作成まで実行できるか

**可能だが、権限設定が必要**。

`--print` モードでもEdit・Bash・Glob等のツールは動作する。ただしデフォルトでは危険な操作（ファイル書き込み、外部コマンド実行等）に都度許可ダイアログが出る。

自動化での対処法:

```powershell
# allowedTools を事前設定（.claude/settings.json）するか
# --permission-mode を指定する
& $claude --print --permission-mode bypassPermissions "..."
```

ただし `bypassPermissions` はDBへの全アクセスや大量削除も通るため、**Maestro Runnerのmax_attempts制限やPAUSEファイル機構と組み合わせること**。

**存在しない**。

Claude Code自体にはファイル監視・自動起動の仕組みはない。Dex提案のMaestro Runner（外部監視プログラム）が必要。PowerShellの `FileSystemWatcher` が最も実装が簡単。

#### Q9. サブエージェントを利用した場合、同じ契約の利用制限をどの程度共有するか

**同じ契約枠を完全に共有する**。

CC本体セッション + Agentツールで起動したサブエージェント = 同一アカウントの利用上限に対してカウントされる。並列で動かすと消費が重なる。これが3サービス体制の最大の利点の一つであり、CCからサブエージェントで全工程を処理する構成は利用制限の面で不利。

---

#### Q11. 誤作動・権限・セキュリティ・利用条件の観点で避けるべき構成

- **`--dangerously-skip-permissions` を常時適用しない**: DB削除・大量削除はKazumaxの承認待ちにすること。
- **`ANTHROPIC_API_KEY` を環境変数に置かない**: API従量課金に切り替わるリスクがある。
- **無限ループ防止**: Dex提案のmax_attempts=3を必ず実装すること。PAUSEファイルによる緊急停止も必須。
- **同時編集排除**: Maestro Runnerは工程ごとに1担当しか動かさないよう設計する。
- **利用規約**: Anthropicの利用規約では自動化・ボット的利用について明示的な禁止はないが、スパム的な過剰実行は規約違反になり得る。作業ごとに合理的な間隔を設けること。

---

#### Q12. 3サービス体制 vs 単一サービス＋サブエージェントのどちらを推奨するか

**3サービス体制を推奨**。理由は以下の3点。

1. **レビューの独立性**: DexがCCの実装意図を知らない状態でレビューするから今のTake2→4の差し戻しで本物のバグが捕捉できている。サブエージェント方式ではオーケストレーターを介して文脈汚染が起きる。
2. **利用制限の分散**: CCだけで大規模改修を行った際に早期に上限へ到達した経験（本ファイルの3節参照）が示すとおり、複数サービスへの分散は実用上重要。
3. **異なるモデルの視点**: Air（Google）、CC（Anthropic）、Dex（OpenAI）という異なるモデルが同じコードを異なる視点で見ることが品質に直結している。
