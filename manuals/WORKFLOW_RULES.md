# Workflow Rules

## 起動時ルール

AIは作業前に以下を確認します。

1. 自分が開いているプロジェクトのルート。
2. `AGENTS.md` の指示。
3. `docs/handoff/CURRENT_STATUS.md` の現在状況。
4. `manuals/AI_TEAM_WORKFLOW.md` と `manuals/WORKFLOW_RULES.md`。
5. 存在する場合は `docs/PROJECT_RULES.md` と `docs/handoff/WORKFLOW_RULES.md`。
6. 必要なhandoffファイル。

## 新規プロジェクトの開始

新規プロジェクトでは、`CURRENT_STATUS.md` を Cycle 1 にして開始します。
別プロジェクトのhandoff履歴、提案、サイクル番号、固有ルールはコピーしません。

## 共通ルールと固有ルールの分離

- 共通ルール: `AGENTS.md`, `CLAUDE.md`, `.cursorrules`, `manuals/`
- 固有ルール: `docs/handoff/CURRENT_STATUS.md`, `docs/PROJECT_RULES.md`, `docs/handoff/WORKFLOW_RULES.md`
- 作業履歴: `docs/handoff/` 配下の各サイクルファイル
- 改善提案: `docs/proposals/`

## 編集前の確認

コードや設定を変更する前に、対象ファイル、目的、影響範囲を短く確認してください。
危険な変更や不明な点がある場合は、実装前にKazumaxへ確認してください。

## 完了時の確認

作業完了時は、何を変更したか、どう確認したか、次に誰の何を待つかを明記してください。

## 次担当トリガー文の必須出力

差し戻し、実装依頼、再レビュー依頼、次担当への正式な作業開始など、Kazumaxが別AIのチャットへそのまま貼る必要がある場面では、担当AIは必ずチャットの最後にコピー用の起動メッセージを単独のテキストブロックで出力してください。

このルールの目的は、Kazumaxが毎回文章を作り直す手間をなくし、Air / Dex / CC の引き継ぎを迷わせないことです。

必須条件:

- テキストブロック内だけをコピーすれば、次担当AIがすぐ動ける内容にする。
- 最初に読むファイル、今回読むべきhandoffファイル、作業内容、保存先、完了時の報告先を入れる。
- 差し戻しの場合は、NG理由の詳細ファイルとTake番号を明記する。
- チャット本文の説明に埋め込まず、必ず独立したコードブロックとして出す。
- すぐ次担当へ渡す必要がない完了報告や、Kazumaxへの説明だけで終わる場合は不要。

例:

```text
まず AGENTS.md、docs/handoff/WORKFLOW_RULES.md、docs/handoff/CURRENT_STATUS.md を読んで、現在地・次担当・完了時ルールを確認してから作業して。
このプロジェクトに docs/PROJECT_RULES.md がある場合は、それも読んで危険領域と検証条件を確認して。

CCへ：
DexがCycle XXのP4レビューを行い、Take2差し戻しになりました。
docs/handoff/P4_Rollback/xxxx.md を読んで、指摘内容を修正し、完了報告を docs/handoff/P3_CC_to_Dex/xxxx_take2.md に保存してください。
```

## サブエージェント（並行処理）の積極活用プロトコル

真の最高効率を叩き出すため、各AIは以下のルールに従ってサブエージェント（内部プロセス）を活用してください。ただし、サブエージェントは調査・読解・確認・レビュー補助のために使うものであり、Air / CC / Dex の役割、実装判断、最終判断、担当AIの責任を代替してはいけません。

1. **並行処理による時短（積極活用）**
   巨大ファイルの解析、最新技術のWebリサーチ、複数ファイルの同時確認、複数観点レビューなど、裏で並行処理させた方が早く安全に終わると判断したタスクについては、AIは自身の権限で利用可能なサブタスク機能・並行レビュー機能を積極的に活用すること。速度だけでなく、リスク、観点数、検証負荷、責任分界も考慮して判断する。
2. **無駄な乱用の禁止（最高効率の追求）**
   サブエージェントの起動にはタイムラグ（オーバーヘッド）が発生するため、メインAI単体で瞬時に処理できる軽微なタスクには使用しないこと。常に「どちらが最速で最高効率か」を計算して判断する。
3. **明確な稼働報告の義務化**
   サブエージェントを利用した際は、通常のトラッキングタグや合図文の邪魔にならない短さで、チャット画面に `[⚡〇〇クルー稼働中: 〇〇の理由で〇〇を並行処理させています]` というタグを出力し、Kazumaxへ「使用した理由と目的」をひと目でわかるように報告すること。

## チーム内でのサブエージェントの呼称規則（〇〇クルー）

Air、Dex、CCといった「実体のあるメインエージェント」と、彼らが呼び出す「裏方の補助役（サブエージェント）」を混同しないよう、Kazumax向けの呼称ではサブエージェントを必ず **「〇〇クルー」** と呼ぶこと。内部メモで `SubReview-A` などのラベルを使ってもよいが、チャット報告やP2/P4本文では「デクスクルーA」のように誰の補助役か分かる名前へそろえる。
- 例：Airが立ち上げたサブエージェント ⇒ **エアクルー** (Air Crew)
- 例：Dexが立ち上げたサブエージェント ⇒ **デクスクルー** (Dex Crew)
- 例：CCが立ち上げたサブエージェント ⇒ **CCクルー** (CC Crew)

## サブレビュー利用時の記録

DexがP2事前監査またはP4レビューでサブエージェントを使った場合、サブエージェントの出力をそのまま最終判断にしないでください。Dexが内容を読み、採用・不採用・保留を判断し、最終的なOK/NGまたはCC向け修正指示として統合します。

P2/P4ファイルには、以下を短く残してください。使わなかった場合でも、高リスクなのに省略したときは理由を残してください。

- なぜサブレビューを使ったか。
- どの観点を任せたか。
- Dexが最終的にどう判断したか。
- 主な指摘ごとの採用 / 不採用 / 保留と理由。
- 残っているリスク。

サブエージェントの意見が割れた場合も、最終判断者はDexです。責任の所在をぼかさないでください。
Kazumaxへのチャット報告には、以下のように短く書いてください。

```text
サブレビュー利用判断: 使用。理由: 金額・Excel・条件保持に触るため、複数視点で確認しました。
```

```text
サブレビュー利用判断: 不使用。理由: 文言修正のみで影響範囲が狭いため、Dex単独で確認しました。
```
## 共通マニュアル更新ルール

共通ルールを変更したAIは、変更作業の最後に必ず同期まで行います。

手順:

1. `C:\Users\kazum\OneDrive\デスクトップ\AI共通マニュアルテンプレ` を更新する。
2. `C:\Users\kazum\OneDrive\デスクトップ\マニュアル同期.ps1` を実行する。
3. 同期先で `AGENTS.md` と `manuals/WORKFLOW_RULES.md` が更新されていることを確認する。
4. 進捗ファイルやプロジェクト固有ルールは上書きしない。

同期するもの:

- `AGENTS.md`
- `CLAUDE.md`
- `.cursorrules`
- `manuals/`

同期しないもの:

- `docs/handoff/CURRENT_STATUS.md`
- `docs/handoff/` のサイクル記録
- `docs/handoff/WORKFLOW_RULES.md`
- `docs/proposals/`
- `docs/PROJECT_RULES.md`
