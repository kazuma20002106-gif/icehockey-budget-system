# AI Team Workflow

このドキュメントは、Kazumax、Air、Dex、CCで作業するための共通フローです。

## 基本方針

P2は「必ずDex」ではありません。
タスクの危険度によって、AirがCCへ直接渡す場合と、Dexが事前監査する場合に分かれます。

## 通常タスクの流れ

軽いUI調整、文言修正、レイアウト調整、モック段階の小さな追加などは、AirがP1とP2相当の実装指示をまとめて作り、CCへ直接渡してよいです。

```text
Kazumax -> Air(P1/P2統合) -> CC(P3) -> Dex(P4レビュー) -> Kazumax
```

1. P1/P2 Air: Kazumaxと壁打ちし、仕様とCC向け実装指示をまとめる。
2. P3 CC: Airの統合指示書に従って実装し、検証結果を報告する。
3. P4 Dex: CCの差分をレビューし、OKまたは差し戻しを判断する。
4. Kazumax: 最終確認し、次サイクルへ進むか決める。

## 危険タスクの流れ

DB、保存処理、金額計算、Excel出力、外部通信、既存データ破壊の可能性がある変更、またはKazumax/Airが不安を感じる変更は、Dexの事前監査を挟みます。

```text
Kazumax -> Air(P1/P2草案) -> Dex(P2事前監査) -> CC(P3) -> Dex(P4レビュー) -> Kazumax
```

1. P1/P2草案 Air: 仕様とCC向け実装案を作る。
2. P2 Dex: Airの草案を読み、リスクを潰した安全なCC向け指示書にする。
3. P3 CC: Dexの指示書に従って実装し、検証結果を報告する。
4. P4 Dex: CCの差分をレビューし、OKまたは差し戻しを判断する。
5. Kazumax: 最終確認し、次サイクルへ進むか決める。

## P2をDexへ回す条件

以下のどれかに当てはまる場合、AirはCCへ直行せずDexへ回してください。

- 保存処理、DB、API、外部通信に触る。
- 金額、集計、帳票、Excel、日付計算に触る。
- 既存データを壊す可能性がある。
- 変更範囲が複数ファイル・複数責務にまたがる。
- 指示内容に曖昧さが残っている。
- Kazumaxが「Dexにも見てもらって」と言った。
- Air自身が少しでも危険だと判断した。

## サブエージェント活用ルール

サブエージェントは、Air / CC / Dex の役割を置き換えるものではありません。必要な場面だけ、担当AIの判断を補強するために使う「専門チェック係」として扱います。

原則として、サブエージェントを使うかどうかは、その作業を担当しているAIが判断します。特にDexは、P2事前監査またはP4レビューに入る時点で、差分の危険度を見てサブレビューを入れるか判断してください。Kazumaxから明示指示がなくても、危険度が高い場合はDex判断でサブレビューを使ってよいです。`サブレビュー利用判断: 使用/不使用。理由: ...` を、作業開始時または完了報告時にKazumaxへ短く伝えてください。

### Dexレビューでの推奨形

Dexが最終レビュアーであることは変えません。サブエージェントは個別観点の調査・確認だけを行い、最終的なOK/NG判定、差し戻し内容、CCへの修正指示はDexが統合して決めます。

```text
SubReview-A: 金額・集計・Excelセル・帳票出力だけを見る
SubReview-B: UI導線・hidden input・条件保持・画面遷移だけを見る
SubReview-C: 既存機能の回帰・差分範囲・未報告ファイル混入だけを見る
Dex: サブレビュー結果を読み、最終P2/P4判断を出す
```

### サブエージェントを使う判断基準

以下のどれかに当てはまる場合、Dexはサブレビューの利用を検討してください。特に金額、DB、Excel、既存データ破壊、外部通信、認証・権限、削除・上書き、データ移行、秘密情報に触る場合は、原則としてサブレビューを使います。使わない場合は、P2/P4ファイルとKazumaxへの報告に理由を明記してください。

- 金額計算、集計、帳票、Excel出力、セル座標に触る。
- DB保存、DB読込、mapper、schema、既存データに触る。
- API、外部通信、認証・権限、課金や外部サービス連携に触る。
- 削除、上書き、データ移行、バックアップ、ロールバック、秘密情報や個人情報に触る。
- hidden input、検索条件、対象データの絞り込み、画面遷移により出力対象が変わる。
- 差分が大きい、または複数責務にまたがる。
- Take2 / Take3 以降で、同じ領域の見落としを防ぎたい。
- Dexひとりのレビューでは観点が多すぎると判断した。

以下の場合は、原則としてサブエージェントを使わなくてよいです。

- 文言修正だけ。
- マニュアルやhandoffの軽微な整理だけ。
- 小さいCSS/UI調整だけ。
- 影響範囲が狭く、Dexひとりで十分に確認できる。

### 汎用例

```text
例1: 金額Excel出力の修正
Dexはサブレビューを入れる。金額・Excel担当、UI条件保持担当、回帰確認担当に分ける。

例2: ボタン文言の変更
Dex単独レビューでよい。サブレビューは使わない。

例3: 検索条件を保持してダウンロードする導線の修正
Dexは必要に応じてサブレビューを入れる。特にhidden input、GET/POSTパラメータ、対象件数の変化を見る担当を置く。
```

### 記録ルール

サブレビューを使った場合、DexのP2/P4ファイルに以下を短く記録してください。サブレビューを使わなかった場合も、Kazumaxへの報告に不使用理由を短く添えてください。高リスクなのに使わなかった場合は、P2/P4ファイルにも省略理由を残してください。

- サブレビューを使った理由。
- サブレビューに割り当てた観点。
- サブレビュー結果をDexがどう統合したか。
- 主な指摘ごとの採用 / 不採用 / 保留と理由。
- 残っているリスクや、Dex単独判断にした理由。

サブエージェントの意見が割れた場合も、最終判断者はDexです。責任の所在をぼかさないでください。
## Air自身による実装（直直し）の特別ルール

Airは原則として企画・設計（Blueprint作成）を担当しますが、「わざわざCCへ引き継ぐまでもない軽微な作業」や「即座に完了できる修正」に限り、**Air自身が直接実装を行うことが許可**されます。

ただし、以下の条件を**厳守**してください。
1. **事前確認の徹底**: 実装を行う前に、必ずKazumaxに「これはAirが直接直してよいか？」と確認を取り、明示的な承認を得ること。
2. **隠蔽の禁止（言語道断）**: Kazumaxに対して「CCに実装させます」と宣言しておきながら、裏でこっそりAir自身が実装する行為は絶対に禁止します。
3. **対象は安全なタスクのみ**: 危険タスク（金額計算、Excel出力、DB変更など）は決してAirが直接実装してはならず、必ずDexの事前監査（P2）とCCによる実装（P3）を通すこと。

## Handoff Protocol

長い指示や報告は、チャットだけでなくファイルに保存します。

推奨フォルダ例:

- `docs/handoff/P1_Air_Blueprint/`
- `docs/handoff/P2_Dex_to_CC/`
- `docs/handoff/P3_CC_to_Dex/`
- `docs/handoff/P4_Dex_Review/`
- `docs/handoff/P4_Rollback/`
- `docs/proposals/`

Airが通常タスクとしてCCへ直接渡す場合も、指示書は `docs/handoff/P1_Air_Blueprint/` などに保存してください。
DexがP2事前監査を行う場合は、CC向け最終指示書を `docs/handoff/P2_Dex_to_CC/` に保存してください。

## 次担当への起動メッセージ

差し戻し、実装依頼、再レビュー依頼など、Kazumaxが別AIを起動する必要がある場合は、作業したAIがそのまま貼れる起動メッセージを必ずチャットへ出します。

```text
まず AGENTS.md、docs/handoff/WORKFLOW_RULES.md、docs/handoff/CURRENT_STATUS.md を読んで、現在地・次担当・完了時ルールを確認してから作業して。
このプロジェクトに docs/PROJECT_RULES.md がある場合は、それも読んで危険領域と検証条件を確認して。

CCへ：
DexがCycle XXのP4レビューを行い、Take2差し戻しになりました。
docs/handoff/P4_Rollback/xxxx.md を読んで、指摘内容を修正し、完了報告を docs/handoff/P3_CC_to_Dex/xxxx_take2.md に保存してください。
```

起動メッセージは、Kazumaxが説明文を足さなくても次担当AIが迷わず動ける粒度にします。差し戻し先、読むファイル、作業内容、保存先を必ず入れてください。

## サイクル番号

サイクル番号はプロジェクトごとに管理します。

- 既存プロジェクト: `docs/handoff/CURRENT_STATUS.md` の Current cycle を読む。
- 新規プロジェクト: Cycle 1 から開始する。
- 別プロジェクトのサイクル番号を引き継がない。

## 進捗管理

現在の状況は必ず `docs/handoff/CURRENT_STATUS.md` にまとめます。
このファイルだけはプロジェクト固有です。共通テンプレからコピーした後、プロジェクト名とサイクル番号を更新してください。
